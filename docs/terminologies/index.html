Core Terminologies & Concepts

Tyxar introduces distinctive terminology that reimagines how developers think about programming constructs. Understanding these foundational concepts is essential for writing effective, expressive Tyxar code.

Umbrellas: Organized Code Structure

Umbrellas are declaration containers that organize and structure your code at the top level. Think of them as organized buckets that group related declarations together—they keep your code dry and protected, much like actual umbrellas protect you from the rain (except these protect you from code confusion).

Umbrellas must be placed at the very beginning of your Tyxar program and serve as the primary mechanism for declaring variables, functions, imports, and other program elements.

Available Umbrellas

Currently, the Variables Umbrella is available for use in all versions of Tyxar. Future versions will introduce additional umbrella types to expand language capabilities:

  • Variables Umbrella (Currently Available) – Declare and initialize variables for use throughout your program
  • Functions Umbrella (Future) – Define reusable functions with parameters and return values
  • Library Import Umbrella (Future) – Import external libraries and modules into your program
  • Abilities Umbrella (Future) – Enable and configure advanced language features and capabilities

Variables Umbrella: Declaring Your Data

The Variables Umbrella allows you to declare all variables used in your program. Each variable declaration follows Tyxar's natural English-like syntax: 

Variables:
    the age is "" as a digit
    the name is "John" as a phrase
    the background is 2
    the calculation is 1+1*2/3^6

Each line within the Variables Umbrella follows the pattern: the [name] is [value] [as a state]

The as a state portion is optional—Tyxar's intelligent type inference will automatically determine the appropriate state if you omit it.

Future Umbrella Examples

To provide a glimpse into Tyxar's future capabilities, here's how functions might be declared in an upcoming Functions Umbrella (not yet available): 

Functions:
    func1: par1, par2
        (...function body...)

States: Tyxar's Take on Types

In Tyxar, we call them States instead of "types" because, well, why stick with convention when you can be delightfully different? States define the fundamental nature of data and what operations can be performed on them. They work identically to types in other languages—Tyxar just gives them a cooler name.

Tyxar provides automatic state inference, meaning the language automatically determines the state of your data based on its value. However, you can also hard-code states explicitly for clarity, documentation, or validation purposes.

Built-in States

Tyxar includes a comprehensive set of built-in states to represent different kinds of data:

Numeric States

  • digit (aka integer) – Whole numbers without decimal points (e.g., 42, -17, 0)
  • decimal (aka float) – Numbers with decimal precision (e.g., 3.14, -2.5, 0.001)

Text States

  • phrase (aka string) – Sequences of characters enclosed in quotes (e.g., "Hello, World!", "Tyxar")
  • letter – A single alphabetic character (e.g., 'a', 'Z')

Special Character State

  • symbol – A non-alphanumeric character (e.g., !, @, #)

Immutability State

  • constant – A value that cannot be modified after initial assignment, even if modification is attempted. Constants provide guarantees that certain values remain unchanged throughout program execution.

State Inference vs. Explicit Declaration

Tyxar intelligently infers states from the values you assign. However, you can explicitly declare the state for added clarity or validation: 

Variables:
    the implicit_age is 25                    (automatically inferred as digit)
    the explicit_age is 25 as a digit         (explicitly declared as digit)
    
    the implicit_pi is 3.14159                (automatically inferred as decimal)
    the explicit_pi is 3.14159 as a decimal   (explicitly declared as decimal)
    
    the implicit_name is "Alice"              (automatically inferred as phrase)
    the explicit_name is "Alice" as a phrase  (explicitly declared as phrase)

Discovering State Information: state id

To programmatically determine or display the state of any piece of data, use the state id command. This is particularly useful for debugging, validation, or when you need to dynamically understand what you're working with: 

write the state id for "hello"           (outputs: phrase)
write the state id for 42                   (outputs: digit)
write the state id for 3.14                 (outputs: decimal)
write the state id for the name             (outputs: phrase, since name = "John")

Best Practices for Umbrellas and States

  • Place Umbrellas at the Top – Always declare your Umbrellas at the beginning of your program for maximum clarity and organization.
  • Use Explicit States When Unclear – While type inference is powerful, explicitly declaring states for complex or non-obvious assignments improves code readability and serves as documentation.
  • Leverage state id for Debugging – When you're uncertain about a value's state or need to understand data flow, use state id to quickly verify.
  • Consider Constants for Fixed Values – Use the constant state for values that should never change, providing compile-time guarantees and reducing runtime errors.
