Language Basics

Annotations and Comments

Annotations, commonly known as comments, are crucial for documenting your code. They enable you to explain your logic, document your decision-making process, and help future readers (including your future self—who will stare at this code in six months wondering why on earth you wrote it that way) understand your reasoning. In Tyxar, comments are completely ignored by the compiler and serve purely as documentation.

Comment Syntax

Tyxar uses a distinctive delimiter-based comment syntax that embraces symmetry and clarity. All annotations must be surrounded by the # character on both sides. Think of it as verbal bookends—your comment gets safely wrapped in protective symbols. This approach works for both single-line and multi-line comments, making it intuitive and consistent regardless of comment length.

The basic structure is: # your comment here #

Single-Line Comments

For brief annotations on a single line, simply wrap your comment with # delimiters. Perfect for those "oh, that's why I did that" moments: 

# This is a single-line comment #

# Initialize the user counter (because negative users would be weird) #
the count is 0

# Tyxar will ignore this entire line (and pretend it never happened) #

Multi-Line Comments

Tyxar's comment syntax naturally extends to multiple lines. Open your comment with #, write multiple lines of documentation, then close with #. Everything between the delimiters is ignored— it's like having a secret conversation with anyone who reads your code later: 

# This is a multi-line comment that spans
several lines of documentation. Tyxar will
ignore all of this text completely, allowing
you to write extensive explanations, documentation,
and reasoning. No judgment here. #

the name is "Alice"

# The following section implements the core
business logic. We use a digit state to ensure
type safety for our calculations. Spent three hours
debugging this. Don't touch it. It works now.
I don't know why, but it does. #
the total is 100

Comment Placement and Best Practices

Comments can appear anywhere in your code, but strategic placement enhances readability:

  • Above code blocks – Explain what the following section does or why it exists (spoiler: sometimes the only explanation is "legacy code")
  • Beside variable declarations – Document the purpose and expected values
  • Before complex logic – Explain non-obvious algorithms or business rules
  • At the top of programs – Provide overview, author information, or usage instructions

Comprehensive Comment Examples

# ========================================
Program Documentation
Author: Your Name
Purpose: Calculate user statistics
Last Updated: April 2026
This took way longer than expected but we got there
======================================== #

Variables:
    # User identification and tracking #
    the user_id is 12345 as a digit
    
    # Full name stored as phrase for display #
    the user_name is "John Doe" as a phrase
    
    # Core calculation variables (don't ask why these numbers, they just work) #
    the total_purchases is 0 as a decimal
    the transaction_count is 0 as a digit

# Main program logic begins here
This section processes all user transactions #

write the user_name
write the user_id

Important Notes on Comments

  • Comments are completely ignored – They have zero impact on program execution, so feel free to vent to your code. It listens without judgment.
  • Nested delimiters not supported – You cannot nest # delimiters within comments. Simple rules, simple life.
  • Whitespace is preserved – Formatting and indentation within comments are preserved as you write them, so you can create formatted documentation.
  • Use comments wisely – Write code that is self-documenting through clear naming. Comments should explain why, not what. If someone has to read your comment to understand what the code does, the code probably needs simplifying.

Variables: Storing Your Data

Variables are containers that store data values throughout your program. Think of them as labeled boxes where you can put things for safekeeping and retrieve them later. In Tyxar, all variables must be declared in the Variables Umbrella, which lives at the very top of your program—keeping everything organized and easy to find.

This approach is refreshingly different from many languages where variables scatter throughout your code like forgotten coffee cups across a desk. In Tyxar, there's one place to look: the Variables Umbrella. Want to know what data your program works with? Check the umbrella. It's beautifully organized chaos prevention.

The Variables Umbrella Declaration

All variables must be declared under the Variables Umbrella at the start of your program. The umbrella heading can take several equivalent forms:

  • Variables:
  • VARIABLES:
  • VAR:

All variable declarations must be indented under this umbrella heading. This visual nesting makes it immediately clear which variables belong together— your code organization gets a promotion from "chaotic" to "intentional."

Variable Declaration Syntax

Variables in Tyxar follow a natural English pattern. Every variable declaration must begin with the keyword the (or The—case doesn't matter here). This mimics natural language and makes your code read more like documentation:

Pattern: the [variable_name] is [value] 

Variables:
    the age is ""
    the name is "Bob"

Variable Naming Conventions

  • Letters, numbers, and underscores only – Variable names can contain these characters, but must start with a letter.
  • Case-sensitive – the_name and The_Name are different variables (which is why clarity beats cleverness every time).
  • Use underscores for spaces – Since spaces aren't allowed, use underscores for multi-word names: the_user_age, the_total_purchases
  • Be descriptive – the_user_id is infinitely better than the_x. Your future self will thank you, and code reviewers won't need therapy.

Specifying States (Optional)

Variables can be assigned specific States using the as a keyword. This is optional—Tyxar can infer states automatically— but explicit states improve code clarity and serve as documentation:

Pattern: the [name] is [value] as a [state]

Comprehensive Variables Example

Variables:
    # Using explicit states for clarity
    the age is "" as a digit
    the name is "John" as a phrase
    the background is 2
    the num is 1+1*2/3^6

More Complex Examples with All States

Variables:
    # Numeric states
    the user_age is 25 as a digit
    the user_balance is 3500.50 as a decimal
    
    # Text states
    the greeting is "Hello, World!" as a phrase
    the initial is 'J' as a letter
    the separator is '|' as a symbol
    
    # Special states
    the max_users is 100 as a constant

State Inference vs. Explicit Declaration

Tyxar is intelligent enough to figure out what state your variables should be. You can omit the as a [state] part and let Tyxar do the heavy lifting: 

Variables:
    # Tyxar automatically figures these out
    the count is 42                 (automatically a digit)
    the price is 19.99              (automatically a decimal)
    the message is "Hello"          (automatically a phrase)
    
    # But being explicit is fine too
    the count is 42 as a digit
    the price is 19.99 as a decimal
    the message is "Hello" as a phrase

Important Variable Notes

  • Umbrella must be at the top – All Variables Umbrellas go at the very beginning of your program, before any other code.
  • Variables can be empty – You can declare a variable with an empty value: the name is "" is perfectly valid. Useful for placeholders.
  • Variables can have expressions – Mathematical calculations are allowed: the total is 1+1*2/3^6 works just fine.
  • Use state id for debugging – If you're unsure about a value's state, use the state id command to check: write the state id for the_variable

Operators

Tyxar supports various operators for performing operations on values.

Arithmetic Operators

VAR:
                        the sum is 2 + 3          # Addition: 5
                        the difference is 10 - 4   # Subtraction: 6
                        the product is 3 * 4       # Multiplication: 12
                        the quotient is 15 / 3     # Division: 5
                        the power is 2 ^ 3         # Exponentiation: 8
                        the remainder is 10 % 3    # Modulo: 1

Comparison Operators

VAR:
                        the isEqual is 5 == 5      # Equal to: true
                        the isNotEqual is 5 != 3   # Not equal to: true
                        the isGreater is 10 > 5    # Greater than: true
                        the isLess is 3 < 5        # Less than: true
                        the isGreaterEqual is 5 >= 5  # Greater than or equal: true
                        the isLessEqual is 3 <= 5     # Less than or equal: true

Logical Operators

VAR:
                        the result1 is true and false   # Logical AND: false
                        the result2 is true or false    # Logical OR: true
                        the result3 is not true         # Logical NOT: false

String Operations

VAR:
                        the greeting is "Hello" + " " + "World"  # Concatenation: "Hello World"
                        the name is "john"
                        the upperName is to adjust the name to CAPS  # Uppercase conversion

Control Flow

Conditionals

Conditional statements allow you to execute different code blocks based on conditions.

If Statements

Use if statements to execute code when a condition is true:

VAR:
                            the age is 18
                            
                            IF the age is greater than 16:
                                write "You can drive"
                            END IF

If-Else Statements

Add an else clause to handle the case when the condition is false:

VAR:
                            the temperature is 25
                            
                            IF the temperature is greater than 20:
                                write "It's warm outside"
                            ELSE:
                                write "It's cool outside"
                            END IF

Else-If Statements

Chain multiple conditions using else-if:

VAR:
                            the score is 85
                            
                            IF the score is greater than 90:
                                write "Grade: A"
                            ELSE IF the score is greater than 80:
                                write "Grade: B"
                            ELSE IF the score is greater than 70:
                                write "Grade: C"
                            ELSE:
                                write "Grade: F"
                            END IF
